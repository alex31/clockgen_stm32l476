MP3 loop refinement
===================

Current state
-------------
- The firmware now uses generated per-asset metadata:
  - `skip_samples`: encoder delay to discard at loop start.
  - `play_samples`: useful decoded PCM span to replay before rewinding.
- Those values are generated by `SOUNDS/generate_mp3_asset.py`.
- This removed the audible silence/padding gap on short loops.
- Remaining defect on short loops: a small tick can still be heard because the
  loop boundary is close in value, but not always close enough in slope.

Why this must stay in the asset pipeline
----------------------------------------
- The embedded code should stay generic and not carry per-track hand-tuned
  constants.
- MP3 is not intrinsically loop-friendly; the right place to compensate is
  asset preparation, not the playback state machine.
- The firmware should only consume a compact, explicit contract:
  `skip_samples` + `play_samples` (+ possibly loop window metadata later).

Next improvement to implement
-----------------------------
1. Improve loop-end search in `generate_mp3_asset.py`.
   The current heuristic already searches near the tail and penalizes value and
   slope mismatch, but it is still simplistic.
2. Search over a longer tail window for short periodic sounds.
   Candidate scoring should consider:
   - sample value discontinuity
   - first-derivative discontinuity
   - short window RMS error between loop head and candidate tail
3. Add an optional very short crossfade at the loop boundary.
   Goal: remove residual clicks caused by slope discontinuity.
   A fade length around 32..128 samples should be enough.
4. Keep the playback engine unchanged if possible.
   Prefer preprocessing the PCM used for MP3 encoding rather than adding more
   logic in the embedded decoder path.

Constraints to preserve
-----------------------
- No per-sample ad hoc constants in `audio.hpp`.
- Metadata must be generated automatically alongside the MP3 asset.
- Long tracks such as `cosmos` and `double_tamponne` must continue to use the
  same generic metadata path as short loops.
- Flash usage is tight; any solution that adds a second playback path or large
  lookup tables should be avoided unless clearly justified.

Suggested validation when revisiting this
-----------------------------------------
- Regenerate `tone500`, `tone500t10`, `sweep`.
- Listen on hardware for:
  - silence gap removal
  - residual click level
  - no change in pitch or loop duration
- Confirm generated `play_samples` remains sensible for long tracks.
